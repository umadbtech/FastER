# API Contract (Feature-Wise, Frontend-Focused)

This document is for frontend/UI teams.

It only covers user-facing features and their API calls.

Frontend env values used by client:
- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`
- `VITE_EVENT_A_ID` (event context for app features)

The key and URL will be shared separately and securely.



The link provided on this page points to the latest consolidated version, which may change as development is still in progress.
Once the architecture is finalized, the updated and finalized details will be organized and published feature-wise in the left-side sections.

json

When a user is created and verified, rows are automatically managed across three main tables via database triggers. Here is the breakdown:
1. auth.users (Managed by Supabase)
This is the core identity table.
When created: A new row is inserted as soon as the signup request is sent.
When verified: The email_confirmed_at (for email) or phone_confirmed_at (for phone) column is updated from NULL to the current timestamp.
2. public.profiles
This table holds user-specific public information (like usernames).
When created: A row is automatically created via a trigger (on_auth_user_created) as soon as the user signs up.
Data: Initially, it only contains the user_id linked to the auth user.
3. public.user_verifications
This table is used by the application logic to quickly check verification status without querying the internal auth schema.
When created: A row is inserted automatically upon signup.
When verified: A trigger (trg_sync_user_verifications_auth_update) detects the change in auth.users and automatically updates email_verified or phone_verified to true in this table.

1) Save Username
Endpoint

Headers
apikey: <SUPABASE_ANON_KEY>

Content-Type: application/json
Request
{
  "username": "JohnDoe99"
}


Notes for frontend
Do not send user_id.

Username length enforced: 3..30.

Upserts public.profiles for current user.

{

401 missing/invalid bearer token

Validation rules

required if array contains self_describe

not future
within last 120 years

Returns activation state + missing gates.
Response (example)
{
  "saved": true,
  "activated": false,
  "status": "onboarding",
  "missing": ["primary_emergency_contact", "terms_acceptance"]
}

Feature: Accept Terms & Conditions (Auto-Activation Aware)
Endpoint
POST /functions/v1/accept-terms
Headers
apikey: <SUPABASE_ANON_KEY>
Authorization: Bearer <ACCESS_TOKEN>
Content-Type: application/json
Request
{}

Notes for frontend
Do not send user_id.
Do not send version from client.
Backend derives:
user from JWT
terms version from server config (app_runtime_config.required_terms_version, fallback v1.0)
What happens on save
Upserts into public.terms_acceptances for current user + required version.
Calls public.maybe_activate_membership().
Returns activation result and missing gates.
Response (example: still onboarding)
{
  "saved": true,
  "terms_version": "v1.0",
  "activated": false,
  "status": "onboarding",
  "missing": ["primary_emergency_contact"]
}

Response (example: activated now)
{
  "saved": true,
  "terms_version": "v1.0",
  "activated": true,
  "status": "active",
  "missing": []
}

Common errors
401 invalid/missing/expired JWT
500 server config issue (if required config is broken)
Frontend usage
Call after user taps “I agree”.
Use returned status and missing to decide whether onboarding is complete immediately.

Activation behavior (important for frontend)
All 4 endpoints above auto-run activation check via maybe_activate_membership().
maybe_activate_membership() evaluates DB truth gates:
username exists
demographics complete
if minor: guardian accepted + guardian consent
primary emergency contact exists (festival-scoped)
required terms accepted
Status behavior
onboarding -> active when all gates pass
never downgrades active -> onboarding
returns:
{
  "activated": true,
  "status": "active",
  "missing": []
}

Accept:
{
  "friendship_id": "<friendship_uuid>",
  "action": "accept"
}

Reject:
{
  "friendship_id": "<friendship_uuid>",
  "action": "reject"
}

Block:
{
  "friendship_id": "<friendship_uuid>",
  "action": "block"
}

Unblock:
{
  "friendship_id": "<friendship_uuid>",
  "action": "unblock"
}

Rules:
only addressee can accept/reject pending
either side can block
only blocker can unblock
Read my friendships
5. friendships?select=*&festival_id=eq.<festival_uuid>&or=(requester_id.eq.<my_user_id>,addressee_id.eq.<my_user_id>)&order=created_at.desc
Realtime subscription (frontend)
subscribe to public.friendships changes filtered by festival_id
on event, process row only if requester_id==me || addressee_id==me
events observed:
send request -> INSERT
accept/reject/block/unblock -> UPDATE
Status behavior summary
pending: waiting for addressee
accepted: friends
rejected: can be re-requested (reopened)
blocked: request attempts return 409 until unblock
